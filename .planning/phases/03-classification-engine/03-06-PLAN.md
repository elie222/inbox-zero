Static rules and learned patterns retain priority. Multi-category messages and other categories proceed through the existing LLM flow unchanged. Implements **CLASS-09**.

<background>
The 30-day production sample (812 messages) showed `CATEGORY_PROMOTIONS → Marketing` precision is 83%. No other Gmail category is clean enough for a shortcut. The hint-route variant (passing categories into the LLM prompt) was deferred — low value relative to prompt complexity, and we already pay for those LLM calls.

`CATEGORY_PURCHASES` is not provisioned for the production account (probed 2026-05-08; only `PROMOTIONS`, `SOCIAL`, `UPDATES`, `FORUMS`, `PERSONAL` exist in `users.labels.list`). The Gmail UI "Purchases" view is a server-side query, not a label, so per-message receipt classification still goes through the LLM.

The Marketing rule has fixed actions (LABEL=Marketing, ARCHIVE — see `apps/web/utils/rule/consts.ts:70`). `choose-args.ts:108` already short-circuits when no `{{template}}` variables exist, so the `ai-actions` LLM call is *already* skipped for Marketing today. CLASS-09's win is purely the `aiChooseRule` skip.

**Reference precedent: the existing Calendar PRESET match.** `match-rules.ts:192-202` already implements an analogous shortcut: when a message is a calendar invite, it pushes a `PRESET` match for `SystemType.CALENDAR` directly — bypassing AI matching for that rule. The Gmail-category shortcut should mirror this pattern exactly (same `MatchReason` shape, same loop position).
</background>

### Task 1 — Add CATEGORY_* constants

**File:** `apps/web/utils/ai/choose-rule/match-rules.ts` (top of file, near other consts)

Add a single non-exported constant for the structural-label allowlist used by the shortcut check:

```ts
// Gmail labels considered "structural" — they don't affect categorization.
// A message is "PROMOTIONS-only" if its labelIds contain CATEGORY_PROMOTIONS
// and no other CATEGORY_*, regardless of these structural labels.
const GMAIL_STRUCTURAL_LABELS = new Set([
  "INBOX",
  "UNREAD",
  "IMPORTANT",
  "STARRED",
  "SENT",
]);

const GMAIL_CATEGORY_PROMOTIONS = "CATEGORY_PROMOTIONS";

function hasOnlyPromotionsCategory(labelIds: string[] | undefined | null): boolean {
  if (!labelIds || labelIds.length === 0) return false;
  let sawPromotions = false;
  for (const id of labelIds) {
    if (id === GMAIL_CATEGORY_PROMOTIONS) {
      sawPromotions = true;
      continue;
    }
    if (id.startsWith("CATEGORY_")) return false; // any other category disqualifies
  }
  return sawPromotions;
}
```

**Why a helper:** the predicate is non-trivial (sawPromotions + reject-other-categories + ignore structural labels by construction) and is unit-testable in isolation. Not exported — internal to `match-rules.ts`.

### Task 2 — Add the shortcut inside `findPotentialMatchingRules`

**File:** `apps/web/utils/ai/choose-rule/match-rules.ts` — inside the rule loop in `findPotentialMatchingRules`, mirroring the Calendar PRESET pattern at lines 190-202.

Insertion point: after the `runOnThreads` guard (line 217) and **before** the learned-pattern check (line 219). This keeps learned-pattern priority — the loop simply considers a different rule per iteration; the Marketing-rule iteration will hit the shortcut, but a different rule's learned-pattern iteration will still match for the same message and override it later via `hasLearnedPatternMatch`.

Wait — re-read. The loop iterates rules one at a time. The Calendar PRESET fires *only* when `rule.systemType === CALENDAR`. We mirror this: fire the Marketing shortcut *only* when `rule.systemType === MARKETING`, AND the message is `CATEGORY_PROMOTIONS`-only. Inside the same iteration we then `continue` so the rule isn't double-processed. The Marketing rule won't appear in `potentialAiMatches`, so `aiChooseRule` is skipped (assuming no other rules' AI conditions need evaluation — which is the existing behavior; learned patterns clear `potentialAiMatches`, but a learned-pattern match on a *different* rule for this message is the only competitor and is handled by `hasLearnedPatternMatch`).

Actual code:

```ts
// Gmail category shortcut — CLASS-09.
// If the message has only CATEGORY_PROMOTIONS (no other CATEGORY_*),
// route directly to the Marketing rule and skip the LLM call.
// Static rules and learned patterns iterate independently and retain priority via
// the existing matches/hasLearnedPatternMatch logic.
const promotionsShortcut =
  rule.systemType === SystemType.MARKETING &&
  hasOnlyPromotionsCategory(message.labelIds);

if (promotionsShortcut) {
  matches.push({
    rule,
    matchReasons: [
      { type: ConditionType.PRESET, systemType: SystemType.MARKETING },
    ],
  });
  continue;
}
```

Place this **after** the calendar block and **after** the runOnThreads guard, **before** the learned-pattern block. This gives the rule a chance to be filtered out by `runOnThreads` (a Marketing thread continuation should still respect that guard) but ensures the shortcut wins over the rule's own learned-pattern/static evaluation for the same iteration.

<risks>

- **Risk: a future rule with `systemType: MARKETING` semantics but custom AI conditions.** Today `MARKETING` is a fixed system rule with no AI instructions; if someone adds AI conditions to it, the shortcut would silently skip them. Mitigation: low concern — Marketing is system-defined and not user-editable per-condition; if this changes, CLASS-09 needs revisit. Captured as a comment in the shortcut block.

- **Risk: Gmail miscategorizing transactional emails as PROMOTIONS.** 17% of PROMOTIONS-tagged messages in our sample were not actually marketing — some were Newsletters that "look promotional." User has explicitly accepted this 83% precision tradeoff; the misroutes are low-harm (Newsletter→Marketing both get archived/labeled, not lost).

- **Risk: future rule reorder might place a learned-pattern rule *after* Marketing, causing Marketing's shortcut to fire before learned-pattern eval on a different rule.** The current loop iterates all rules and only filters at the `hasLearnedPatternMatch` stage, so order doesn't actually matter — the shortcut adds to `matches`, learned-pattern eval on a different rule also adds to `matches`, and `filterMultipleSystemRules` resolves multi-system collisions downstream. Confirmed safe by re-reading lines 288-376.

- **Risk: `matchReasons` schema drift.** The PRESET match shape is already used by Calendar; we're reusing the same shape. No schema change.

</risks>
