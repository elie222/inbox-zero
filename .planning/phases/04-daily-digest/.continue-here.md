<completed_work>
- **Bug #1 — digest arriving at 00:00 ET instead of 09:00 ET.** Root cause: commit `3a34259c5` reimplemented schedule advancement in `runDailyDigest` and the in-transaction `Schedule.update` wasn't persisting (`updatedAt` advanced, `nextOccurrenceAt`/`lastOccurrenceAt` stayed frozen). Couldn't pin the exact Prisma/pg-adapter root cause without runtime instrumentation. Per user direction, dropped the reimplementation entirely rather than re-debug: removed the `digestSchedule.nextOccurrenceAt` filter from `runDailyDigest`'s findMany and removed the `schedule.update` step from the transaction. The cron container now fires `/api/cron/digest` once a day at 09:00 ET using busybox-compatible `date -D '%H:%M' -d '09:00' +%s`.
- **Bug #2 — Notifications bucket always 0.** Root cause: `BUCKET_BY_RULE` keyed on rule name `"Notifications"` (plural); actual rule is `"Notification"` (singular). ## Critical Anti-Patterns (do NOT repeat these)
- **Reimplementing upstream logic inside fork-only code.** Commit `3a34259c5` duplicated upstream's `getDigestScheduleProgression`/`schedule.update` flow inside `runDailyDigest` instead of reusing the existing path. The duplicate path had a latent bug; the upstream path is presumably fine. Future digest changes: extend the rendering layer (V2 props, buckets) but route scheduling through upstream's pattern or skip scheduling entirely as we now do.
- **Name-keyed matching for user-renameable entities.** `BUCKET_BY_RULE` keyed on rule `name` silently broke when the rule got renamed `Notifications → Notification`. Prefer `systemType` (enum, immutable) for anything that needs to survive renames.
