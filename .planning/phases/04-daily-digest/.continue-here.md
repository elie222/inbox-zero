Couldn't pin the exact Prisma/pg-adapter root cause without runtime instrumentation. Per user direction, dropped the reimplementation entirely rather than re-debug: removed the `digestSchedule.nextOccurrenceAt` filter from `runDailyDigest`'s findMany and removed the `schedule.update` step from the transaction. <decisions_made>
- **Branch + PR instead of direct-to-main.** Even on a solo fork, the digest path auto-deploys from main and a digest-timing regression already burned us; a PR gives a clean rollback point.
- **Drop the schedule coupling rather than fix it.** User's principle: "the more we diverge the harder it gets to merge." The schedule-advance reimplementation in `runDailyDigest` was the divergence; deleting it reduces fork footprint and the single-user fork doesn't need per-account scheduling.
- **systemType-first bucketing, name fallback for Urgent/Uncertain.** Promoting Urgent/Uncertain to new `SystemType` enum values would mean a Prisma migration and further upstream divergence ‚Äî not worth it for a single-user system where a silently-missing section is quickly noticeable (matches the 2026-05-08 reasoning).
- **Grey out + tooltip rather than hide.** Keeps the upstream UI component intact and makes the override visible/explainable, per user request.

## Critical Anti-Patterns (do NOT repeat these)
- **Reimplementing upstream logic inside fork-only code.** Commit `3a34259c5` duplicated upstream's `getDigestScheduleProgression`/`schedule.update` flow inside `runDailyDigest` instead of reusing the existing path. The duplicate path had a latent bug; the upstream path is presumably fine. Future digest changes: extend the rendering layer (V2 props, buckets) but route scheduling through upstream's pattern or skip scheduling entirely as we now do.
- **Name-keyed matching for user-renameable entities.** `BUCKET_BY_RULE` keyed on rule `name` silently broke when the rule got renamed `Notifications ‚Üí Notification`. Prefer `systemType` (enum, immutable) for anything that needs to survive renames.
