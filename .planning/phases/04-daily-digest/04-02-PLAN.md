Purpose: Without this table, idempotency cannot be enforced and a stuck/restarted cron could send the same digest twice.

Add the following model immediately after the existing `model Digest { ... }` block (or `model DigestItem` if it sits between — keep all digest-related models contiguous):

```prisma
model DigestSend {
  id                 String       @id @default(cuid())
  createdAt          DateTime     @default(now())
  updatedAt          DateTime     @updatedAt
  emailAccount       EmailAccount @relation(fields: [emailAccountId], references: [id], onDelete: Cascade)
  emailAccountId     String
  /// ET calendar date the digest covers (UTC midnight of that ET date — see utils/digest/today-et.ts).
  date               DateTime     @db.Date
  sentAt             DateTime
  /// Resend message id returned by the SDK; null if delivery failed before Resend ack.
  resendMessageId    String?
  itemCount          Int
  /// Aggregate Sonnet token usage for this send (cost monitoring).
  inputTokens        Int?
  outputTokens       Int?
  modelUsed          String?
  /// Snapshot of generated narrativeBody for replay/debug. Truncate to 4000 chars at insert time.
  narrativeSnapshot  String?
  /// Array of Digest.id values batched into this send (audit trail).
  digestIds          String[]

  @@unique([emailAccountId, date])
  @@index([emailAccountId])
  @@index([sentAt])
}
```

Also add the back-relation to `EmailAccount`. Locate `model EmailAccount { ... }` and add this line near the other relation arrays (e.g. next to `digests Digest[]` or similar):
```prisma
  digestSends DigestSend[]
```
If the EmailAccount model does not currently have a `Digest[]` relation array, that's fine — just add the `digestSends DigestSend[]` line. Do not modify any other field.
